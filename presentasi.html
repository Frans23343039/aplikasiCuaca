Tinjauan Pustaka

Model Client-Server

Menurut Tanenbaum & Steen (2006), arsitektur client-server merupakan paradigma komputasi terdistribusi di mana client melakukan request dan server memberikan response. Model ini memisahkan logika presentasi (client) dari logika bisnis dan data (server).

Protokol HTTP dan REST

HTTP (Hypertext Transfer Protocol) adalah protokol aplikasi yang stateless dan bekerja pada layer 7 (aplikasi). REST (Representational State Transfer) menyediakan arsitektur untuk merancang layanan web yang scalable (Fielding, 2000).

Konkurensi dan Multi-threading

Server modern menggunakan multi-threading atau event-loop untuk menangani multiple clients. Node.js menggunakan event-driven non-blocking I/O, sementara Express.js menyediakan middleware untuk request handling (Node.js Foundation, 2024).

Keamanan dalam Client-Server

Keamanan mencakup: (1) Enkripsi data (HTTPS/TLS), (2) Autentikasi dan autorisasi, (3) Input validation, (4) Rate limiting (Kurose & Ross, 2021).

Aspek Keamanan

3 Ancaman Keamanan Utama

1. Man-in-the-Middle (MITM) Attack

  • Ancaman: Penyerang mengintersepsi komunikasi HTTP
  • Solusi pada Aplikasi Besar: HTTPS/TLS encryption
  • Implementasi Kami (sederhana): HTTP di localhost (assume trusted network)

2. API Key Exposure

  • Ancaman: API keys tertanam di kode atau JavaScript frontend
  • Solusi pada Aplikasi Besar: Backend proxy, environment variables, API key rotation
  • Implementasi Kami: API keys di backend (node modules), tidak exposed ke client

3. Injection Attack (Input Validation)

  • Ancaman: User input tidak divalidasi → XSS, SQL Injection
  • Solusi pada Aplikasi Besar: Input sanitization, parameterized queries
  • Implementasi Kami: encodeURIComponent() untuk query parameter, tidak menerima input berbahaya
// Contoh: Validasi input di backend if (!req.query.address) { return res.send({ error: 'Kamu harus memasukan lokasi' }); } const encodedAddress = encodeURIComponent(req.query.address);

Analisis Keterbatasan

  • Caching: Tidak ada cache → setiap request ke API eksternal
  • Error Recovery: Sederhana, tidak ada retry mechanism
  • Scalability: Single server, tidak bisa scale horizontally
  • Database: Tidak ada persistence layer (data di memori/JSON)

Pembahasan

Apakah Program Berhasil Mereplikasi Fitur Inti?

Ya. Program berhasil mereplikasi fitur inti aplikasi cuaca:

  • ✓ Request kota dari client
  • ✓ Geocoding (konversi kota → koordinat)
  • ✓ Fetch data cuaca dari API eksternal
  • ✓ Tampilkan data ke client dalam format JSON/HTML
  • ✓ Handle concurrent requests dengan event loop

Tantangan Teknis yang Dihadapi

  1. Asynchronous Callback: Nested callbacks (callback hell) → solusi: menggunakan Promise atau async/await
  2. API Rate Limiting: Weatherstack memiliki batas request → solusi: implement caching atau backoff strategy
  3. Error Handling: Multiple APIs dapat gagal → solusi: implement fallback dan retry logic
  4. CORS (Cross-Origin): Browser melarang request lintas domain → solusi: gunakan CORS middleware

Perbandingan dengan Teori

Implementasi kami sesuai dengan teori client-server karena:

  • Pemisahan antara presentation layer (client) dan business logic layer (server)
  • Menggunakan HTTP protocol yang stateless
  • Event-driven architecture yang efficient untuk concurrent requests
  • REST API principles untuk resource-based design

Kesimpulan dan Saran

Kesimpulan Utama

  1. Arsitektur Terbukti: Client-server architecture dengan Express.js dan HTTP/REST efektif untuk aplikasi cuaca real-time
  2. Event Loop Adequate: Node.js event loop mampu menangani concurrent requests tanpa threading kompleks
  3. Integrasi API: Integrasi multiple external APIs (Weatherstack, Positionstack, Mediastack) berjalan stabil
  4. Keamanan Dasar: Implementasi keamanan dasar (input validation, backend API keys) sudah diterapkan
  5. Scalability Limitations: Single-server setup memiliki keterbatasan untuk scale production

Saran Pengembangan Lebih Lanjut

  • Caching Layer: Implementasikan Redis untuk cache data cuaca (reduce API calls)
  • Database Persistence: Gunakan MongoDB atau PostgreSQL untuk history cuaca
  • Autentikasi User: Implement JWT untuk user authentication dan personalization
  • Microservices: Pisahkan modul geocoding, weather, dan news menjadi microservices terpisah
  • HTTPS/Security: Deploy dengan HTTPS dan implement rate limiting
  • Frontend Enhancement: Gunakan React atau Vue.js untuk UI yang lebih interaktif
  • Real-time Updates: Implement WebSocket (Socket.IO) untuk weather updates yang push-based
